Technology

Compressed data that remains computationally useful.

ChromoFold combines navigable compressed representations with fused GPU consumers. Values are decoded only where they are used, normally into registers, instead of reconstructing an entire tensor first.

Conventional

Materialize, then compute

  1. Read compressed data.
  2. Allocate a full output tensor.
  3. Decode every element.
  4. Read that tensor again in the consumer.
ChromoFold

Navigate, decode, consume

  1. Keep compact pages resident.
  2. Locate the needed block.
  3. Decode a value inside the consumer.
  4. Use it immediately and discard the register value.

Why it can win

Trade arithmetic for memory.

Modern inference frequently waits on VRAM capacity or bandwidth. A small number of integer operations can be cheaper than loading a much larger materialized representation from HBM.

saved allocation + saved memory traffic > decode overhead

This is the operating condition, not a universal guarantee.

System layers

One idea, several serving surfaces.

KV cache

Compressed pages with a small active tail for recent tokens.

Weights

Fused matrix consumers avoid a full decompressed working set.

Shared prefixes

Common prompt state is referenced once instead of copied per request.

Searchable state

Rank, access and pattern operations work over compact GPU-resident indexes.

Where it can lose

Decode is not free.

Compute-bound kernels

Extra decode instructions may reduce throughput when arithmetic is already saturated.

Tiny intermediates

Eliminating a small buffer may not recover the integration cost.

Poor locality

Random block access, divergence or metadata pressure can erase bandwidth savings.
